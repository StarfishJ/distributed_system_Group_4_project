# Test results & report assets

Generated by **`client_part2`** and optional scripts. Paths are relative to the **project root** (or `client/client_part2` working directory when running the JAR from there — the client writes `results/` under the **current working directory**).

## Default files (single `results/` folder)

| File | Description |
|------|-------------|
| `per_message_metrics.csv` | Per-message latency rows (used for **P50 / P95 / P99** in the printed summary) |
| `throughput_over_time.csv` | 10-second bucket throughput (chart input) |
| `metrics_last.json` | Pretty-printed **`GET /metrics`** body from the **last** run |

## Tagged runs (`--results-tag`)

Avoid overwriting baseline vs stress outputs:

```text
java -jar ... --results-tag baseline500k http://<ALB> 500000
java -jar ... --results-tag stress1m http://<ALB> 1000000
```

Creates **`results/baseline500k/`** and **`results/stress1m/`** each with their own CSV + `metrics_last.json`.

## Report screenshots (example layout)

For **PerformanceReport.md** you can keep PNGs in subfolders, e.g.:

| Folder | Typical contents |
|--------|------------------|
| `50w/` | Baseline run: `Core.png` (**GET /metrics** Core 1–4), `metrics.png` (client terminal / summary if captured), `Analytical.png`, `MQ.png`, `CPU & Memory.png` |
| `100w/` | Stress run: same filenames (note: `CPU & memory.png` may differ by casing) |

**Naming hint:** **`Core.png`** = Core queries in the terminal; **`metrics.png`** = broader terminal capture (often includes client summary / metrics block). Align captions with **`PerformanceReport.md`**.

## Throughput chart (optional)

```bash
cd results
python generate_throughput_chart.py
# or: python generate_throughput_chart.py path/to/throughput_over_time.csv
```

Requires Python 3 + `matplotlib` (`pip install matplotlib`). Output: `throughput_chart.png`.

## Other files

- `generate_throughput_chart.py` — builds the line chart from CSV buckets  
- `profile.html` — profiling output if generated
